# Internationalization

## Goals

Let a single Blume project serve docs in multiple languages: locale-aware
routing, per-locale navigation and search, translated UI chrome, and correct
SEO (`hreflang`, localized canonicals, sitemap). The authoring path stays
convention-first — drop translated files into the right place and Blume wires up
routing, the language switcher, and fallbacks with zero boilerplate, matching
the zero-config ethos in `00-vision.md`.

Blume is build-time and static-first, so the design must work in pure static
output (no cookie-based locale detection on the critical path). Astro ships
native i18n routing; Blume should lean on it for URL plumbing and own the
docs-specific layer (per-locale nav, UI dictionaries, switcher, search).

Non-goals:

- Runtime/automatic machine translation. Translation is an authoring concern;
  Blume integrates with external tooling but doesn't translate content itself.
- A general-purpose app-i18n library. Like Fumadocs, Blume handles docs content,
  routing, and its own UI strings — not arbitrary application strings.

## Prior art

### Fumadocs

Headless and explicit. A shared config defines the languages; a middleware does
locale routing; the loader is made locale-aware; UI strings come from a
translations object passed to a provider. Fumadocs is candid that it "is not a
full-powered i18n library."

```ts
// lib/i18n.ts
import { defineI18n } from "fumadocs-core/i18n";
export const i18n = defineI18n({
  defaultLanguage: "en",
  languages: ["en", "cn", "vi"],
  hideLocale: "default-locale",  // always | default-locale | never
  fallbackLanguage: "en",        // null disables fallback
  // parser: "dir" (en/page.md) | "dot" (page.cn.md)
});
```

Notable mechanics worth stealing:

- **`hideLocale` modes** — `never` (default, always prefix), `default-locale`
  (only the default is unprefixed), `always` (cookie-based; flagged as bad for
  static sites). Blume is static-first, so `always` is out; `default-locale` is
  the sensible default.
- **`fallbackLanguage`** — missing translations fall back to a chosen locale
  (defaults to the default language); `null` disables it. Shared,
  locale-agnostic files use a `$` marker: `my-page.$.md`, `meta.$.json`.
- **`parser`** — `dir` (locale directories) or `dot` (`page.fr.md` filename
  suffix). Two ergonomics for two situations: full vs. sparse translation.
- **Per-locale page tree** — `source.getPageTree(lang)` returns nav for one
  locale; `loader({ i18n })` makes the source locale-aware.
- **UI strings** — `i18n.translations().extend(uiTranslations()).add({...})`
  with official language packs; passed via `i18nProvider(translations, lang)`.
  `<html lang={lang}>` comes from the `[lang]` route segment.

### Mintlify

Config-driven and convention-light. `docs.json` carries a
`navigation.languages[]` array; each entry is a full, independent navigation
tree tagged with an ISO 639-1 `language`, one marked `default: true`.

```json
{
  "navigation": {
    "languages": [
      { "language": "en", "default": true, "groups": [{ "group": "Getting started", "pages": ["index"] }] },
      { "language": "es", "groups": [{ "group": "Comenzando", "pages": ["es/index"] }] }
    ]
  }
}
```

- Translated files live in ISO 639-1 directories (`es/index.mdx`) mirroring the
  default structure; the default language sits at the root.
- Each language replicates the nav independently and **may diverge** (different
  groups/labels/order per locale).
- The language switcher is auto-generated from the `languages` array; per-locale
  `footer`/`navbar`/`banner` overrides exist, with a `global` object for shared
  cross-language anchors.
- Hard rule: **never reuse the same page path across languages** ("undefined
  behavior").
- Translation is offloaded to agents/CLIs (Locadex, Lingo.dev); the guidance is
  to translate prose and code comments but not code itself.

### Takeaway for Blume

Take Fumadocs' precise primitives (`hideLocale`/`fallbackLanguage`/`parser`,
per-locale page tree, UI dictionaries with built-in packs) and Mintlify's
convention-first authoring (locale directories, auto-generated switcher,
per-locale nav that may diverge). Implement it on **Astro's native i18n** so
Blume doesn't reinvent locale routing — consistent with the "Astro is the
runtime; Blume is the docs product" north-star.

## Where i18n touches Blume today

Nothing is locale-aware yet. The seams:

| Concern | Location | Today | Change |
| --- | --- | --- | --- |
| Route mapping | `core/content.ts:43` `mapRoute()` | no locale notion | extract locale, strip from segments, re-prefix route |
| Discovery | `core/content.ts:152` `discoverContent()` | flat scan | tag each entry with a locale (dir/dot parser) |
| Page model | `core/types.ts:64` `PageRecord` | no `locale` | add `locale`; `id` becomes locale-qualified |
| Config | `core/schema.ts` top-level `blumeConfigSchema` | no `i18n` block | add `i18n` schema |
| Nav model | `core/types.ts:114` `Navigation` | single global nav | per-locale navigation |
| Nav builder | `core/navigation.ts` `buildNavigation()` | one tree | filter by locale → one tree per locale |
| Manifest | `core/manifest.ts` `RouteManifestEntry` | no `locale` | add `locale`; carry alternates |
| Runtime data | `astro/generate.ts:271` `buildRuntimeData()` | one nav payload | per-locale nav + per-page alternates |
| Catch-all page | `astro/templates.ts:559` `[...slug].astro` | locale-blind params | resolve locale; pick locale's nav; emit alternates |
| SEO | `astro/templates.ts:674` canonical/OG block | single canonical | locale canonical + `hreflang` + `x-default` |
| HTML lang | `components/layout/RootLayout.astro` `<html lang="en">` | hardcoded `en` | `<html lang={locale}>` |
| Astro config | `astro/templates.ts` `astroConfigTemplate()` | no `i18n` option | emit Astro `i18n` (locales, routing, fallback) |
| Search | `search/documents.ts` `buildSearchDocuments()` | global index | per-locale index / locale facet |
| UI strings | theme components (below) | hardcoded English | locale dictionaries |

## Configuration

A new top-level `i18n` block, opt-in (absent ⇒ today's single-locale behavior).

```ts
import { defineConfig } from "blume";

export default defineConfig({
  i18n: {
    defaultLocale: "en",
    locales: [
      { code: "en", label: "English" },
      { code: "fr", label: "Français" },
      { code: "ja", label: "日本語", dir: "ltr" },
    ],
    // "dir": en/page.mdx | "dot": page.fr.mdx
    parser: "dir",
    // Hide the prefix for the default locale only (/, /fr/…). Static-safe.
    hideDefaultLocalePrefix: true,
    // Missing translation falls back to this locale's page; null disables.
    fallbackLocale: "en",
  },
});
```

- `locales[]` carries display metadata (`label`, optional `dir` for RTL) so the
  switcher and `<html dir>` need no extra config.
- `parser` mirrors Fumadocs: `dir` (default — full translations, mirrors
  Mintlify and Astro) vs `dot` (sparse translations colocated with originals).
- `hideDefaultLocalePrefix` maps to Astro's `prefixDefaultLocale`. We omit
  Fumadocs' cookie-based `always` mode — incompatible with static output.

## Content conventions

**`dir` parser (default):**

```txt
docs/
  index.mdx            → /            (default locale, prefix hidden)
  guides/quickstart.mdx→ /guides/quickstart
  fr/
    index.mdx          → /fr
    guides/quickstart.mdx → /fr/guides/quickstart
  ja/
    index.mdx          → /ja
```

The default locale lives at the root (prefix hidden when configured); other
locales are top-level directories named by `code`. `mapRoute` detects a leading
segment that matches a configured locale, records it on the `PageRecord`, and
removes it before the usual segment processing — then re-prefixes the route
(unless it's the hidden default).

**`dot` parser:**

```txt
docs/
  guides/quickstart.mdx     → /guides/quickstart   (default)
  guides/quickstart.fr.mdx  → /fr/guides/quickstart
```

**Shared, locale-agnostic files** use a `$` marker (Fumadocs' convention):
`changelog.$.mdx` and `_meta.$.json` apply to every locale, avoiding duplication
for content that doesn't need translating.

## Routing on Astro's native i18n

The generated `astroConfigTemplate()` emits an `i18n` block instead of Blume
hand-rolling locale routing:

```js
i18n: {
  defaultLocale: "en",
  locales: ["en", "fr", "ja"],
  routing: { prefixDefaultLocale: false /* from hideDefaultLocalePrefix */ },
  fallback: { fr: "en", ja: "en" },   // from fallbackLocale
}
```

Blume continues to drive `getStaticPaths` from its own manifest (the catch-all
already does this at `astro/templates.ts:652`), now emitting one path per
`(locale, route)` pair. Astro's i18n gives us locale-aware helpers
(`getRelativeLocaleUrl`) and built-in fallback routing for free; Blume owns the
nav/SEO/switcher semantics on top. Locale fallback is resolved at scan time
(Blume materializes a fallback route entry) so static output is fully
pre-rendered with no client redirect.

## Per-locale navigation

`Navigation` becomes per-locale. `buildNavigation()` runs once per locale over
that locale's pages, so each language gets an independent sidebar/tabs that may
diverge (Mintlify-style). Folder-meta (`_meta.*`) resolves per locale, with
`_meta.$.json` as the shared fallback.

Runtime data (`buildRuntimeData()`) changes shape from a single `navigation` to
`navigation: Record<locale, Navigation>`, plus per-route `alternates` (the set
of locales a given logical page exists in) for the switcher and `hreflang`. The
catch-all selects `data.navigation[locale]` from the resolved page locale.

## UI strings and dictionaries

The core theme has ~20 hardcoded English strings that must move into locale
dictionaries. Known locations from the audit:

- `components/layout/RootLayout.astro` — "On this page", "Last updated on",
  "Previous", "Next".
- `components/layout/PageActions.astro` — "Edit this page on GitHub", "Scroll to
  top", "Give feedback", "Copy as Markdown", "Ask AI about this page", "Send".
- `components/layout/Search.astro` — "Search", "Search docs", "Search
  documentation…", "No results found.".
- `components/islands/ask-ai.tsx` — "Ask AI", "Ask a question…", "Send", error
  strings.

Plan:

- A typed dictionary (`UIStrings`) with a built-in `en` baseline and shipped
  packs for common locales (Starlight and Fumadocs both ship built-in UI
  translations — we should match that so adopters get a translated chrome out of
  the box).
- User overrides merge on top, via `i18n.ui` in config or an `i18n/<locale>.json`
  file; unknown keys per locale fall back to the default locale, then `en`.
- The active dictionary is threaded into components through the generated runtime
  data (server-rendered chrome) and into islands via props (so `ask-ai.tsx` and
  search receive their strings without bundling every locale).

## SEO

- `<html lang={locale}>` and `dir` from the locale config (RTL support).
- `hreflang` alternate `<link>`s for every locale a page exists in, plus an
  `x-default` pointing at the default locale — built from the per-route
  `alternates` set. Threaded through the SEO block at `astro/templates.ts:674`
  and `RootLayout.astro`.
- Per-locale canonical URLs (locale-prefixed unless hidden default).
- Localized sitemap entries and `inLanguage` in JSON-LD (`seo/jsonld.ts`).
- OG image route keyed per locale.

## Search

- `SearchDocument` gains a `locale` field; the default Orama index is partitioned
  per locale (or a single index with a `locale` filter) so results match the
  active language.
- The search island filters to the current locale and can offer an "all
  languages" toggle.
- Hosted providers (Algolia/Typesense/Orama Cloud) get `locale` as a facet,
  mirroring the per-provider client wiring in `astro/templates.ts`.

## Language switcher

A new theme component, auto-generated from `i18n.locales` (Mintlify-style, no
manual config). For the current page it uses the per-route `alternates` to link
the equivalent page in each locale; when a translation is missing it links the
locale home (or the fallback page) and visually marks it. Lives in the header
near search; overridable like any other component (`05-customization.md`).

## Migration

`blume migrate mintlify` (`08-roadmap.md` M8) should map
`docs.json` → `navigation.languages[]` into Blume's locale directories +
`i18n.locales`, including the `default: true` → `defaultLocale` mapping and
per-language nav. This makes Mintlify's most painful hand-config a one-command
import.

## Risks and open questions

- **Default-locale URL shape** — root vs. always-prefixed. `hideDefaultLocalePrefix`
  defaults to hiding it (cleanest URLs), but mixing hidden default + prefixed
  others needs careful canonical/`hreflang` handling to avoid duplicate-content
  signals.
- **Divergent nav vs. shared structure** — Mintlify allows fully independent
  per-locale nav; that's flexible but easy to desync. Consider a "mirror the
  default unless overridden" default to reduce drift.
- **Partial translations** — fallback routing must be unambiguous in static
  output: do we render the fallback content at the localized URL (good for SEO,
  but shows untranslated text) or redirect? Leaning toward render-fallback +
  `hreflang` honesty.
- **`id` uniqueness** — making `PageRecord.id` locale-qualified touches the
  manifest, search keys, and `eject`; coordinate with the
  `source:ref` id change proposed in `22-content-sources.md` so ids get one
  composite scheme, not two.
- **RTL** — `dir` per locale affects layout CSS, not just `<html dir>`; the
  default theme needs an RTL pass.
- **UI pack maintenance** — shipping built-in translations is an ongoing
  community-maintained surface; start with a small, high-quality set.
- **Astro i18n fit** — confirm Astro's `fallback` + manual `getStaticPaths`
  compose cleanly (no double routing) before committing to native i18n over a
  fully Blume-owned scheme.

## Phasing

1. **Config + model.** `i18n` schema, `PageRecord.locale`, `dir`-parser detection
   in `mapRoute`/`discoverContent`, locale-qualified ids. No UI yet; routes just
   gain prefixes. Single-locale projects unaffected.
2. **Routing + nav.** Emit Astro `i18n`, per-locale `getStaticPaths`, per-locale
   `buildNavigation`, runtime data reshape, fallback materialization.
3. **UI dictionaries.** Extract hardcoded strings, ship `en` + a starter pack,
   user overrides, thread into components/islands.
4. **SEO + switcher.** `<html lang/dir>`, `hreflang`/`x-default`, localized
   canonicals + sitemap, the language-switcher component.
5. **Search + `dot` parser + shared `$` files.**
6. **Mintlify migration mapping, RTL polish, docs.**

Roughly ~1.5–2 weeks for a solid first cut; phases 1–2 carry the architectural
risk, 3 is broad-but-mechanical, 4–6 are additive.
